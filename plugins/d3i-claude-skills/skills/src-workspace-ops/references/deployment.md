# Deployment

## How It Actually Runs on SRC

The "Next for data donation" catalog item deploys via Ansible. The Ansible playbook lives in `d3i-infra/researchcloud-items` at `playbooks/deploy-next.yaml`. During provisioning, the playbook:

1. Clones `d3i-infra/mono`
2. Runs `docker build` in `mono/core/` to produce the `self-d3i:latest` image
3. Generates `/docker-compose.yaml` from its template
4. Starts the containers with `docker compose`

After provisioning completes, the mono source is no longer on the workspace — only the built Docker image and the generated compose file remain.

### Filesystem Layout

```
/
├── docker-compose.yaml          # Ansible-managed, defines app + db services
├── ca.crt                       # CA cert for DB TLS
├── server.crt, server.key       # PostgreSQL TLS certs
├── rsc/                         # SRC platform files
└── home/<user>/
    ├── data/                    # External storage mount root
    │   ├── datasets/            # (empty by default)
    │   ├── <volume-name>/       # Named storage volume (e.g., qself2026)
    │   └── volume_2/            # Fluentd log collection volume
    │       ├── docker.log*      # Application/container logs (rotated, gzipped)
    │       ├── nginx.log*       # Nginx access/error logs (rotated, gzipped)
    │       └── system.log*      # System logs (rotated, gzipped)
    └── local-share/
```

### Docker Compose (`/docker-compose.yaml`)

Generated by Ansible from the template in `d3i-infra/researchcloud-items` (`playbooks/deploy-next.yaml`). Two services:

```
services:
  app:
    image: self-d3i:latest        # Built from source during provisioning
    container_name: self-d3i
    ports: 8000:8000              # Nginx proxies to this
    volumes:
      - app_data:/tmp             # STATIC_PATH for uploaded assets
      - /ca.crt:/ca.crt           # DB TLS CA cert
    logging: fluentd → 172.17.0.1:24224
    depends_on: db

  db:
    image: postgres:15
    container_name: db-next
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - /server.crt, /server.key  # TLS certs
    command: starts postgres with SSL enabled
```

### Docker Compose `name:` Field

The original Ansible template used `version: "3"` (deprecated) and had no `name:` field. This caused `docker compose` to fail with "project name must not be empty" when run from `/`, because the project name is derived from the directory name (which is empty at `/`).

**Fix (applied via PR to researchcloud-items):** Replace `version: "3"` with `name: "<app_name>"` in the Ansible template. For workspaces created after this fix, `COMPOSE_PROJECT_NAME` is no longer needed.

**For existing workspaces (created before the fix):** Manually edit `/docker-compose.yaml` — replace the `version: "3"` line with `name: "next"` (or whatever project name is appropriate). Alternatively, continue using `COMPOSE_PROJECT_NAME=next` as a workaround.

### Docker Details

- Docker requires **sudo** on SRC workspaces
- The app image (`self-d3i:latest`) is built from source (`d3i-infra/mono`) during Ansible provisioning
- Container names: `self-d3i` (app), `db-next` (postgres)
- Logging goes to fluentd at `172.17.0.1:24224`, which writes rotated logs to `~/data/volume_2/`

### Nginx

Nginx is SRC-managed. Config lives in:
- `/etc/nginx/nginx.conf` — main config
- `/etc/nginx/conf.d/ssl_main.conf` — TLS/reverse proxy config
- `/etc/nginx/conf.d/map-upgrade.conf` — WebSocket upgrade mapping
- `/etc/nginx/app-location-conf.d/authentication.conf` — SRAM auth config

Nginx logs go to syslog (`/dev/log`, facility local7, tag nginx) and are also collected by fluentd to `~/data/volume_2/nginx.log*`.

### Log Locations

| Log | Location | Format |
|-----|----------|--------|
| App (mono) logs | `~/data/volume_2/docker.log*` | Rotated, gzipped daily |
| Nginx logs | `~/data/volume_2/nginx.log*` | Rotated, gzipped daily |
| System logs | `~/data/volume_2/system.log*` | Rotated, gzipped daily |
| Live app logs | `sudo docker logs self-d3i` | Current container only |
| Live DB logs | `sudo docker logs db-next` | Current container only |
| SRC deployment logs | Download from SRC portal | Actions menu |

## Common Operations

### All docker commands require sudo

```bash
sudo docker ps
sudo docker compose -f /docker-compose.yaml ps
sudo docker logs self-d3i --tail=100
sudo docker logs db-next --tail=100
```

**Critical:** On workspaces created before the `name:` field fix (see above), `docker compose` from `/` fails because the project name (derived from directory) is empty. Use one of these approaches:

```bash
# Option 1: If docker-compose.yaml has a `name:` field (new workspaces or manually patched):
sudo docker compose -f /docker-compose.yaml <command>

# Option 2: If docker-compose.yaml still has `version: "3"` (old workspaces):
sudo COMPOSE_PROJECT_NAME=next docker compose -f /docker-compose.yaml <command>
```

### Restarting the Application

```bash
# If docker-compose.yaml has `name:` field:
sudo docker compose -f /docker-compose.yaml restart app

# Or to fully recreate:
sudo docker compose -f /docker-compose.yaml down
sudo docker compose -f /docker-compose.yaml up -d

# If on an old workspace without `name:` field, prefix with COMPOSE_PROJECT_NAME=next
```

### Checking Application Status

```bash
sudo docker ps                                        # what's running
sudo docker logs self-d3i --tail=100                  # recent app logs
sudo docker logs db-next --tail=20                    # recent DB logs
curl -I http://localhost:8000                          # app responding?
```

### Database Operations

```bash
# Connect to PostgreSQL
sudo docker exec -it db-next psql -U my_user -d my_database

# Dump database (backup to external storage)
sudo docker exec db-next pg_dump -U my_user my_database | gzip > ~/data/<volume>/db-backup-$(date +%Y%m%d).sql.gz

# Restore
zcat ~/data/<volume>/backup.sql.gz | sudo docker exec -i db-next psql -U my_user my_database
```

### Checking Nginx

```bash
sudo systemctl status nginx
sudo nginx -t
cat /etc/nginx/conf.d/ssl_main.conf
```

### Viewing Historical Logs

```bash
# Recent docker/app logs
zcat ~/data/volume_2/docker.log.20260322_1.log.gz | tail -50

# Search for errors across log history
zgrep -i error ~/data/volume_2/docker.log.20260322*.gz
```

### Disk Usage

Boot disk is only **15 GB**. Monitor closely:
```bash
df -h /
sudo du -sh /var/lib/docker/* 2>/dev/null | sort -rh | head -10
```

## Environment Discovery (for unfamiliar workspaces)

When SSHing into any SRC workspace for the first time:

```bash
# What's running
sudo docker ps

# Compose config
sudo cat /docker-compose.yaml

# App config (env vars)
sudo docker inspect self-d3i --format '{{range .Config.Env}}{{println .}}{{end}}' | sort

# Nginx config
cat /etc/nginx/conf.d/ssl_main.conf

# Storage mounts
ls ~/data/
df -h

# System resources
free -h
df -h /
```
